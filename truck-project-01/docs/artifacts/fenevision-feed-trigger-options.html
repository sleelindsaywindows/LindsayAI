Option A

Scheduled Pull

A background job queries FeneVision on a fixed schedule (e.g. every morning at 5am). Orders load automatically before Joseph arrives.

⏰ Cron / Scheduler

macOS launchd or Windows Task Scheduler triggers a Python script at 5am daily

→

🔌 FeneVision Query

Script connects via VPN, runs stored procedure for today's route orders

→

💾 Save to file

Results written to fenevision_latest.json in the app data dir

→

📊 App loads

Joseph opens app, finds orders already loaded from last pull

State: App reads fenevision_latest.json on startup. If file is missing or stale (>24h), shows warning. Pull logs written to fenevision_pull.log.

✓ Pros

  • Zero manual step for Joseph — data is just there
  • Pull happens before anyone is in the office
  • Easy to audit (log file)

✗ Cons

  • Requires a machine to be on and on VPN at 5am
  • Stale if orders change after pull time
  • More complex setup (OS scheduler config)

State: No background process. FeneVision credentials stored in .env. VPN status checked before button enabled. Pull result cached in session state only (gone on refresh).

✓ Pros

  • Always pulls latest — no staleness risk
  • No OS scheduler setup needed
  • Joseph controls exactly when data is fresh

✗ Cons

  • Someone must remember to click it
  • Adds a manual step to morning workflow
  • Pull happens during work hours (FeneVision load)
Option C — Recommended

Both: Scheduled + Manual Refresh

Scheduled pull loads data automatically overnight. Manual "Refresh" button lets Joseph re-pull if orders changed. Best of both.

⏰ 5am Scheduled

Background script pulls overnight, writes to cache file

→

📊 App opens

Joseph sees orders already loaded, timestamp shown

→

↻ Manual refresh

If orders changed since 5am, Joseph clicks Refresh to re-pull

→

📋 Diff shown

App highlights what changed: new stops, cancels, priority bumps

After manual Refresh — change summary:

+2 new Valor Communities of GA, BFS Hillsborough

Why this is the right default: Scheduled pull = zero effort for Joseph most mornings. Manual refresh = escape hatch when things change (cancellations, last-minute adds). The diff view is the key feature — it tells him exactly what changed so he doesn't re-plan from scratch.

Recommendation: Start with Manual Only

Build Option B (manual button) first. It's simpler, no scheduler config needed, and proves the FeneVision connection works. Add the scheduled pull (Option A) as a follow-on once Joseph has used manual for a few weeks and trusts the data. The diff view is the real value-add — build it alongside the manual pull.

Open Questions

Need answers before finalizing the spec.

Q1 — VPN access: Does the machine running the app have persistent VPN access to Lindsay's network, or does someone have to connect manually each morning? (This determines if scheduled pull is practical.)
Q2 — FeneVision connection method: Is there an existing ODBC/SQL connection string for FeneVision? Or does the data pull need to go through an API/export endpoint? Who at Lindsay can provide credentials?
Q3 — Route date: Does Joseph always pull today's routes, or does he sometimes plan a day or two ahead? (Affects the date filter in the stored procedure.)
Q4 — Which routes to pull: GA routes only? Or should the feed scope be configurable (GA + NC + other plants later)? The MO interplant route is already excluded — any others?
Q5 — IIS deployment: Once on IIS, does the FeneVision connection work from the server machine, or does each user's browser need to be on VPN? (IIS-hosted Streamlit runs server-side, so VPN needs to be on the server.)
